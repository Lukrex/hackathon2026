About the platform

This is not a brochure site pretending to be a workflow. It is the working system zero one hundred uses to intake requests, route them internally, assign experts, and close the loop only when delivery is confirmed.

Why it exists

Email threads, chat fragments, and spreadsheets break down once multiple roles need to review the same request. This app keeps one record for the request, the people involved, the current status, and the final outcome.

Where it is useful right now

  • Requests need structured intake instead of a vague message in a chat.
  • Internal review must be split between global admins and category-scoped workers.
  • Expert selection should be informed by skills, languages, reputation, and workload.
  • The team needs proof of who owned the request and when it was actually completed.

What the system includes

The current app is already opinionated about how work should move. It is not a generic task board. It models a real intake-to-delivery process with distinct responsibilities.

In scope

  • Request intake with categories, skills, languages, and due dates.
  • Review notes, prioritization, and internal routing.
  • Expert recommendations and assignment tracking.
  • Request chat and internal chat channels.
  • Requester-confirmed completion with expert credit.

What it is not

  • Not an auto-assignment engine that removes human review.
  • Not a public marketplace where every user sees every request.
  • Not a generic CRM replacing all community operations.
  • Not driven by the old Node files still sitting in the repository.
